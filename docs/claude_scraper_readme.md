# Claude Chat Scraper - JSON Output Structure

This document describes the structure of the JSON file generated by the Claude chat scraper.

## Overview

The scraper captures Claude.ai shared conversation pages and outputs a structured JSON file containing the conversation metadata, all turns (user and assistant messages), and artifact versions if present.

## Top-Level Structure

```json
{
  "url": "string",
  "layout": "string",
  "title": "string | null",
  "shared_by": "string | null",
  "turns_count": "integer",
  "turns": [],
  "versions": []
}
```

### Fields

| Field | Type | Description |
|-------|------|-------------|
| `url` | string | The original URL of the scraped conversation |
| `layout` | string | Page layout type: `"one-column"` (chat only) or `"two-column"` (chat + artifact preview pane) |
| `title` | string \| null | Conversation title from the page header |
| `shared_by` | string \| null | Username of the person who shared the conversation |
| `turns_count` | integer | Total number of conversational turns (user + assistant messages) |
| `turns` | array | Array of turn objects representing the conversation flow |
| `versions` | array | Array of artifact version snapshots (only populated for two-column layouts) |

---

## Turn Object Structure

Each turn represents either a user message or an assistant (LLM) response.

### Common Fields (All Turns)

```json
{
  "index": 1,
  "role": "user" | "llm",
  "warning": "string | null",
  "plain_text": "string | null",
  "plain_text_md": "string | null",
  "code": [],
  "analysis":[],
  "aggregated_text": "string | null",
  "thinking": null
}
```

| Field | Type | Description |
|-------|------|-------------|
| `index` | integer | Sequential turn number (1-indexed) |
| `role` | string | Either `"user"` or `"llm"` |
| `warning` | string \| null | Warning/notice text displayed above the message (if any) |
| `plain_text` | string \| null | Plain text version of the message content |
| `plain_text_md` | string \| null | Markdown-formatted version of the message content (assistant only) |
| `code` | array | Array of code artifacts/blocks (see below) |
| `analysis` | array | Array of analysis artifacts/blocks (see below) |
| `aggregated_text` | string \| null | Full text with code artifacts expanded inline on the plain_text|
| `thinking` | object \| null | Extended thinking block (assistant only, see below) |

---

## User Turn Example

```json
{
  "index": 1,
  "role": "user",
  "warning": null,
  "plain_text": "please give me a table on anything",
  "plain_text_md": null,
  "code": [],
  "analysis":[],
  "aggregated_text": "please give me a table on anything",
  "thinking": null
}
```

**Note:** User turns typically have:
- `plain_text_md` = `null` (only assistant messages have markdown)
- `code` = `[]` (empty)
- `thinking` = `null`

## Assistant (LLM) Turn Structure

Assistant turns contain additional fields for thinking blocks and code artifacts.

### Thinking Block

When Claude uses extended thinking, the `thinking` object captures this:

```json
{
  "meta": {
    "header": "Thinking",
    "time": "2s, 15ms"
  },
  "content": "Plain text of thinking content...",
  "content_md": "# Thinking\n\nMarkdown formatted thinking..."
}
```

| Field | Type | Description |
|-------|------|-------------|
| `meta.header` | string | Header text (usually "Thinking") |
| `meta.time` | string | Time spent thinking (e.g., "1m, 50s") |
| `content` | string | Plain text thinking content |
| `content_md` | string | Markdown-formatted thinking content |

### Code Artifacts

The `code` array contains objects for each artifact/code block in the response:

```json
{
  "id": 1,
  "meta": {
    "title": "Avatar Editor Web Component",
    "version": "Version 2",
    "labels": "React",
    "smallline": "React · Version 2"
  },
  "preview_header": "Avatar Editor Web Component",
  "right_title":'v2',
  "right_content": "import React from 'react';\n...",
  "span": [245, 278]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | integer | Artifact number within this turn (1-indexed) |
| `meta.title` | string | Artifact title |
| `meta.version` | string | Version label (if versioned) |
| `meta.labels` | string | Technology/type label (e.g., "React", "HTML", "Python") |
| `meta.smallline` | string | Full metadata line as displayed |
| `preview_header` | string | Header text shown in the artifact preview |
| `right_title` | string | Header text shown on the right|
| `right_content` | string | Full code/content from the artifact preview pane |
| `span` | [int, int] | Character position `[start, end]` where the artifact label appears in `plain_text` |

### Text Variants

Assistant turns provide three text representations:

1. **`plain_text`**: Plain text with artifacts replaced by labels like `[Avatar Editor Web Component · Version 2]`
2. **`plain_text_md`**: Markdown-formatted text with artifacts replaced by labels
3. **`aggregated_text`**: Full expansion where each code label is replaced with:
   ```
   [Title · Version]: <full code content>
   ```
and the analysis block also have its full content

**Example:**

If `plain_text` contains:
```
View analysis:
Here's the component: [Avatar Editor · Version 1]
```

Then `aggregated_text` contains:
```
View anlaysis:

To get the output of...

Here's the component:

[Avatar Editor · Version 1]: import React from 'react';
function AvatarEditor() { ... }
export default AvatarEditor;
```

## Version Captures

For two-column layouts (pages with artifact previews), the `versions` array captures snapshots of each artifact version:

```json
{
  "label": "Version 1",
  "content_text": "Full code content of this version..."
}
```

| Field | Type | Description |
|-------|------|-------------|
| `label` | string | Version label (e.g., "Version 1", "Version 2") |
| `content_text` | string | Complete code/content of this version |

**Note:** Version captures are scraped by:
1. Clicking the version dropdown in the artifact pane
2. Selecting each version sequentially
3. Capturing the displayed content

## Usage Notes

### Working with the Data

**Get all user messages:**
```python
user_messages = [turn for turn in data['turns'] if turn['role'] == 'user']
```

**Get all code artifacts:**
```python
all_artifacts = []
for turn in data['turns']:
    if turn['role'] == 'llm' and turn['code']:
        all_artifacts.extend(turn['code'])
```

**Find turns with extended thinking:**
```python
thinking_turns = [turn for turn in data['turns'] 
                  if turn['thinking'] is not None]
```

**Extract final code from artifacts:**
```python
for turn in data['turns']:
    for artifact in turn.get('code', []):
        title = artifact['meta']['title']
        code = artifact['right_content']
        print(f"{title}:\n{code}\n")
```

### Text Format Recommendations

- Use **`plain_text`** for simple text processing, search, or display
- Use **`plain_text_md`** for rendering in markdown viewers
- Use **`aggregated_text`** when you need the full conversation including complete code artifacts and analysis details
- Use **`code[].right_content`** when extracting artifacts individually
- You could add the content in **`turn['thinking']`** to the **`aggregated_text`** if you want everything generated by LLM

## Known limitations

- **Version dropdown can fail to open**  
  In some Claude layouts, the right-pane “versions” menu (`aria-haspopup="menu"`) does not reliably mount or respond to clicks, so capturing alternate versions may intermittently fail.

- **DOM vs. rendered UI mismatches**  
  The HTML snapshot can differ from what’s actually rendered/interactive. In those cases a selector that looks valid in the HTML may not be clickable in the live UI, leading to missed clicks or wrong targets.

- **Images are not retrievable**  
  Clicking **Show image** does not expose the underlying image source in a way the scraper can capture, so embedded images are currently skipped.
